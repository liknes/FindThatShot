# Changelog

All notable changes to **Find That Shot** are recorded in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **Global map browse view (Catalog → Browse on map…).** A new non-modal window turns location into a primary navigation axis — the "where did I shoot that?" entry point for a geo-heavy drone archive. Every geotagged clip is plotted as a **clustered** marker on the same bundled-offline Leaflet stack the per-clip preview uses (now joined by the **Leaflet.markercluster 1.5.3** plugin, likewise embedded in the assembly and inlined into the page so clustering works with no CDN; only the OpenStreetMap *tiles* need a connection). The map drives the catalog grid three ways: **clicking a cluster** zooms into it *and* scopes the main grid to exactly those clips; the **Filter grid to this view** toolbar button scopes the grid to every clip currently inside the map viewport (pan/zoom to a region, then filter to it); and **clicking a single marker** fills a side panel with that clip's thumbnail, location, coordinates, rating, and online state, and selects it in the grid — with **Show in grid** to bring it forward and **Play in app** to open it in the in-app player. A **scope toggle** flips the plotted set between the *whole archive* (default) and only the clips matching the *current grid filters*. When the grid is scoped from the map, a clearable **"Map selection: N clips"** pill appears in the status bar; clicking it — or changing any normal filter — drops the scoping and restores the full result set. The window is read-only over the catalog: it reuses the lightweight projection the grid already builds (a new `ISearchService.GetGeotaggedClipsAsync` returning a slim `MapClipPoint` per clip — no tag/moment graph — so plotting thousands of points stays cheap) and **no source video file is read, moved, or modified**. Under the hood this adds the `MapClipPoint` projection and a `SearchQuery.VideoIds` id-set filter (the grid-scoping mechanism, sharing `SearchService`'s extracted filter pipeline so the map and grid stay in sync), a `GlobalMapView` WebView2 control (clustering + marker-select / cluster-filter `postMessage` bridges, with map-fit computed from raw point bounds to survive chunked cluster loading), and the `MapBrowseWindow` / `MapBrowseViewModel`, all wired through DI and shown as a single reusable instance (re-invoking the menu brings it forward and refreshes its current-filter scope).

## [0.10.0] - 2026-06-04

### Added

- **AI auto-tagging & natural-language search (opt-in, fully local).** Lights up the long-dormant `AiTagSuggestions` table with a local **CLIP** model (image + text encoders, run on the CPU via ONNX Runtime). Rather than scoring the single thumbnail — a poor signal for a whole clip — the pass **samples several frames across each clip** — duration-proportional, ~one frame every N seconds (default 20s) clamped to a configurable min/max, via the same read-only ffmpeg pipeline that makes thumbnails, scaled+center-cropped to raw RGB — and **max-pools** tag/search scores across them, so "does this clip ever show snow?" is answered by the best frame, not the average. Two things fall out of one model: (1) **auto-tag suggestions** — each clip's frames are scored against a scene/subject vocabulary (sea, fog, ships, birds, beach, forest, mountains, city, snow, sunset, …) using prompt-ensembled text embeddings, and above-threshold labels become `AiTagSuggestion` rows surfaced in a new **Catalog → Review AI suggestions…** window as accept/reject chips (accept promotes it to a real tag; reject is *remembered* so a re-run won't re-propose it); and (2) **natural-language search** — toggle **View → Search by description (AI)** and type something like *"drone shot over snowy mountains at sunset"* to rank clips by CLIP similarity (best-matching frame timestamps are tracked too, for future jump-to-the-moment). Nothing is ever auto-applied, and **source files are only ever read** — the feature stores its own per-frame and pooled per-clip embeddings in the catalog. The whole subsystem is **off by default** and gated behind **Settings → AI tagging**: a master toggle, model setup (the ~340 MB CLIP ONNX bundle is **not shipped** — produce it once with the new `scripts/export-clip-onnx.py` exporter, which downloads public `openai/clip-vit-base-patch32` weights and assembles the exact bundle into `tools/models/clip-vit-b32`; or drop one in via **Model folder**; or self-host a `.zip` and use the in-app **Download model** button. `AiModelProvider` resolves a configured folder → a bundled `tools/models/clip-vit-b32` → managed app-data, and the build/publish copy steps ship `tools/models/` alongside the exe just like FFmpeg/mpv), tuning for the seconds-per-frame sampling interval, min/max frames per clip, and the suggestion confidence threshold, and a **Clear AI data** button that purges embeddings + pending suggestions (accepted tags and source files are kept), and both an incremental **Generate AI tags…** (only clips lacking embeddings) and a **Re-score all clips with AI…** pass (idempotent re-run after changing sampling settings/vocabulary/model — keeps accepted/rejected tags, refreshes pending ones). A scoring pass runs **off the UI thread**, so the app stays fully usable while it works, and a **status-bar pill** tracks it end-to-end: it spins *"AI tagging"* while running (click it to open the review window and start triaging clips already scored), then flips to a green, clickable *"Review N suggestion(s)"* the moment the pass finishes so the result is impossible to miss — clicking opens the review queue and dismisses the pill. The candidate vocabulary was broadened with coastal/drone/aerial and terrain/urban subjects (coastline, cliffs, marina, lighthouse, reef, kayak, canyon, glacier, skyscrapers, wind turbines, …), and an opt-out **adaptive per-label threshold** learns each label's confidence cutoff from your accept/reject history (Youden's-J separation with a min-sample gate and a clamped band) so suggestions sharpen toward your footage over time — the CLIP weights are never modified. Under the hood this adds `AiClipEmbedding` + `AiFrameEmbedding` entities and a `State` lifecycle on `AiTagSuggestion` (with EF configs, a migration, and indexes), a data-driven CLIP ONNX engine (`ClipOnnxModel` + a faithful CLIP BPE `ClipTokenizer` + `ClipModelManifest`), an ffmpeg `IFrameSampler`, an `IAiModelProvider` (presence / download / drop-in resolution + cached model loading), and the `IAiTaggingService` / `IAiSuggestionService` / `IAiSemanticSearchService` orchestration plus the `AiReviewWindow` / `AiReviewViewModel` review UI, all wired through DI and surfaced via opt-in settings.

- **Moments — timestamped in/out markers & sub-clips ("the shot," not just "the file").** Until now tagging, rating, and notes were all whole-file, but the app is literally called *Find That Shot* — so the natural unit is a **moment**, not a clip. You can now mark a timestamped in/out range inside a clip and give it its own **label, rating (0–5), notes, and tags**, fully independent of the parent file's metadata. **Capture** happens live in **Review mode**: press **I** to drop an in-point at the playhead and **O** to set the out-point and create the moment (the new **Mark in** / **Mark out** transport buttons do the same), or press **O** alone to record a single-timestamp marker; a status line in the transport bar tracks the pending in-point. A **thumbnail is grabbed at the in-point automatically** (FFmpeg `-ss` at that exact second, cached under a dedicated `Thumbnails/Moments/` subfolder so it never collides with the parent clip's thumbnail). New moments appear in a **MOMENTS** list in the right-hand editor — select one to edit its label/rating/notes/tags in a compact inline editor, **Jump to** seeks the player straight to that timestamp, and **Delete** removes just the moment. Grid cards show a small **moment-count badge**. A dedicated **Catalog → Find moments…** window searches moments across the whole archive (matching label, notes, and tags, with a minimum-rating filter and an online-only toggle); each result shows the in-point thumbnail, time range, parent file, and tags, and **Jump to** opens the parent clip in Review mode and seeks directly to the moment (e.g. *jump to 00:01:32*). Moments are persisted in the catalog and, when sidecars are enabled, **travel with the footage** — the sidecar schema is bumped to `findthatshot/v2` with a `moments` array (each carrying its in/out seconds, label, rating, notes, and tags), older `v1` sidecars are still read, and write paths that aren't moment-aware (e.g. bulk edits) **preserve** any moments already on disk rather than dropping them. On a fresh import the scanner rehydrates moments from the sidecar (creating rows, linking tags through the same get-or-create path as clip tags, and grabbing in-point thumbnails best-effort), import-only so a rescan never re-creates moments you've since pruned. Catalog removal (single/multiple, offline, by-root) now also prunes the matching moment thumbnails from the cache. As with all curation, capturing or deleting a moment only ever edits the catalog and the thumbnail cache — **the source video file is never modified**. Under the hood this adds `VideoMoment` + `MomentTag` entities (with EF configurations, cascade deletes, indexes, and a migration), `IMomentService`/`MomentService` for CRUD + tag attach + search, timestamp-based thumbnail generation on `IThumbnailService`, and the `MomentViewModel` / `MomentSearchViewModel` / `MomentSearchWindow` UI, all wired through DI.

## [0.9.7] - 2026-06-03

### Added

- **Sidecars are now read back on import — your curation rehydrates a freshly-rebuilt catalog.** Until now sidecar files (`.findthatshot.json`) were write-only: the app kept them in sync next to each video but never read them. As of this release, when the scanner **imports a brand-new clip** it looks for a sidecar beside the file and, if present, restores the curation captured there — **rating, status, notes, location text, context, the parsed folder/shoot date, and all tags** — onto the new catalog record. This closes the round-trip: delete and rebuild your catalog (or move the library to a new machine / fresh install) and your ratings, statuses, notes and tags come back from the files on disk instead of being lost. It is deliberately **import-only** — a rescan of a clip that's *already* in the catalog never re-applies the sidecar, so the live database stays the source of truth and your in-app edits are never clobbered by an older file. Tags are matched/created by their natural `(Name, Type)` key (defaulting to *Subject* when a type is missing), and brand-new tags discovered in sidecars are created on the fly. Reads are best-effort and resilient: a corrupt, partial, locked, or offline-drive sidecar logs a warning and is skipped (degrades to "no sidecar") rather than failing the scan. Under the hood this adds `ISidecarService.TryReadAsync` returning a transport-agnostic `SidecarData`/`SidecarTagData`, consumed by `VideoScannerService` via new `ApplySidecarScalars` / `ApplySidecarTagsAsync` helpers; the tag get-or-create is serialised behind a small semaphore so parallel scan workers can't race two inserts of the same tag past the unique `(Name, Type)` index.

- **Drop loose files (not just folders) onto the sidebar to add their folder as a root.** The FOLDERS drop target now also accepts dropped *files* — each contributes its containing folder as a root candidate — so dragging a mix of folders and loose clips (e.g. the contents of a folder) registers the right roots instead of silently discarding the files. The whole dropped set is then **collapsed by ancestry**: a parent folder supersedes any of its subfolders in the same drop, and any candidate already covered by an existing registered root (the root itself *or* an ancestor of it, compared case-insensitively on normalised full paths) is skipped — not just exact-path duplicates. The result is one clean root per intended location rather than redundant nested roots. New `MainViewModel` helpers `NormalizeFolderPath` / `IsStrictDescendant` back the comparison; the **Add folder…** picker shares the same `AddRootFoldersByPathsAsync` path, so it benefits too.

### Changed

- **The player seek bar is now click-anywhere and spans the full width.** Previously you had to land precisely on the thin 4px accent line to scrub, and the time readouts were pinned to fixed 60px columns. The slider template now carries a transparent full-height backdrop across the entire control (and inside the progress repeat-button), so a click **anywhere in the seek band** jumps the playhead to that point — the click's X position is mapped directly to the seek value. The row was also relaid so the current-time and duration readouts auto-size and flank a full-width slider that stretches across the whole player panel.

- **Refreshed splash screen artwork** (`SplashScreen.png`), and removed the splash window's `DropShadowEffect`. The shadow forced WPF to rasterise the whole splash subtree to an intermediate 1x bitmap before scaling, which made the artwork look soft on high-DPI displays (and it was clipped by the window edge anyway); dropping it keeps the PNG pixel-sharp. The stale `Splash-Screen-0.9.2.png` asset was removed.

### Fixed

- **Clicking the seek track no longer snaps the playhead back to where it was.** The 250ms position poller could fire in the brief window after a click — before the engine reported the new position — and reset the slider to the old spot, so clicks appeared to "jump back". The seek guard is now raised on mouse-**down** (covering plain track clicks, which never raise the thumb's `DragStarted`), the new position is **awaited** before the guard is released, and a short 350ms cooldown suppresses the poller until the engine catches up.

## [0.9.5] - 2026-06-03

### Added

- **Duplicate finder (Catalog → Find duplicates…).** A new non-modal window surfaces sets of catalog entries that are almost certainly the same footage living in more than one place — the classic case being the same clip copied to a backup drive. This is "Tier 1" detection: clips are grouped purely by a **metadata fingerprint** the scanner already stored — **exact file size + duration (rounded to the second) + resolution** — so the scan is effectively instant, needs no new migration, and works for **offline** clips too (no source file is ever opened or hashed). Each duplicate set is shown as a card (e.g. *"2 copies · 1.2 GB each · 1920×1080 · 0:45"*, with the reclaimable bytes called out) listing its members with thumbnail, filename, folder path, size/resolution/codec/camera, and an **Online/Offline** badge. The finder flags one member as the **Suggested keep** (preferring online, then better-curated by rating/tag count, then the oldest catalog id) but never auto-selects a destructive action: you tick exactly which redundant copies to forget, with **Select all redundant** / **Clear** helpers and a live running total (*"N selected · X GB of disk no longer cataloged"*). **Remove selected from database…** confirms first (and explicitly warns if you've selected *every* copy in a set), then removes only those catalog rows + their cached thumbnails via the existing `IVideoLibraryService` — **source video files on disk are never moved, renamed, or deleted.** Removing copies re-runs the scan so resolved sets drop off, and raises a `CatalogChanged` event the main window listens for to refresh its grid. Under the hood this adds a new `IDuplicateDetectionService` (Core interface + Data implementation that narrows to size-colliding rows in SQL, then fingerprints in memory to keep the exact-second rounding out of the query), a `DuplicatesViewModel`, and a `DuplicatesWindow`, all wired through DI and shown as a single reusable instance (re-invoking the menu brings it forward). Groups are ordered by reclaimable savings so the biggest wins surface first.

- **The main window now remembers its size, position, and maximized state across launches.** Previously the window always reopened at the fixed XAML default (1400×880, centered), discarding any resize/move/maximize from the last session — annoying on a multi-monitor setup or when you'd dialled in a size. The window now persists its placement to `settings.json` on close (`AppSettings.WindowLeft` / `WindowTop` / `WindowWidth` / `WindowHeight` / `WindowMaximized`, nullable so an un-saved value falls back to the centered default) and restores it before first paint, so it reopens exactly where you left it with no flash through the default position. The geometry saved is always the **normal-mode** bounds (read from `Window.RestoreBounds`, not the live size) plus a separate maximized flag, so a window closed while maximized reopens maximized *and* un-maximizes back to your last floating size. Restore is defensive: size and position are validated independently (a stale/garbage size falls back to the XAML default while still honouring a saved position), and the position is checked against the **current** virtual-screen bounds via a new `IsOnScreen` helper — a window saved on a monitor that's since been unplugged or rearranged falls back to CenterScreen rather than opening off-screen where you can't grab it. Persistence is best-effort (run off the UI sync-context and briefly awaited on close so the file is flushed before exit, without deadlocking the dispatcher) and mirrors the existing sidebar-width / panel-collapse persistence pattern (`MainViewModel.InitialWindowPlacement` / `PersistWindowStateAsync`). As a side fix, `SidebarSavedSearchesExpanded` is now round-tripped through the settings merge alongside the other sidebar flags (it was being silently dropped on reload).

- **Catalog statistics dashboard (Catalog → Catalog statistics…).** A new non-modal, **read-only** window summarises the whole catalog at a glance so you can see the shape of your archive — and your curation progress — without running a stack of searches. The top **OVERVIEW** strip shows headline cards: total clips (with online/offline split), combined file size, combined runtime, distinct folders (and how many are registered roots), distinct camera models, tag count, the share of clips that are **reviewed**, and the share that are **geotagged**. Below it, a responsive grid of **BREAKDOWN** cards renders proportional bars for clips **by status** (workflow order, Unreviewed → … → Archive), **by rating** (★ buckets), **by resolution** (8K / 4K / 1440p / 1080p / 720p / SD, bucketed on the long edge so portrait clips classify correctly), **by year** (using the parsed folder shoot-date, falling back to the file's modified date), and the **top cameras / codecs / tags** (capped to the busiest 12 each). Every bar shows its count and its share of the catalog. The numbers are computed on demand with plain `COUNT` / `SUM` / `GROUP BY` queries against the SQLite catalog via a new `ICatalogStatisticsService` (Core interface + Data implementation), reshaped for the UI by a new `CatalogStatsViewModel`; an empty catalog shows a friendly "add a folder and scan" hint and a **Refresh** button recomputes the snapshot. As with everything in the app, building the dashboard never reads, moves, or modifies any source video file — it only queries the catalog database. The window is resolved from DI and shown as a single reusable instance (re-invoking the menu brings it forward).

- **Catalog backups now also snapshot `settings.json`.** Each catalog backup writes a companion copy of your `settings.json` into the same Backups folder as `settings-<stamp>.json`, sharing the exact timestamp of its `catalog-<stamp>.db` so the pair is obvious on disk — so a backup now captures your configuration (storage/FFmpeg paths, pinned-tag hotkeys, sidebar layout, sidecar/playback/map toggles, scan settings, …) and not just the database. The settings copy is **best-effort**: it's taken right after the catalog copy succeeds and any failure (or a not-yet-created `settings.json`) is logged and skipped without failing the catalog backup, since the database is the part that matters. Retention is unchanged and the two stay in lockstep — when `PruneAsync` trims an old `catalog-<stamp>.db` beyond the keep count it also deletes the matching `settings-<stamp>.json` (older catalog backups that predate this feature simply have no sidecar to remove). Restore is unaffected: it still stages only the catalog database, so the saved `settings.json` is there to copy back by hand if you ever need it rather than silently overwriting your live configuration.

- **Saved searches (Smart Collections) in the sidebar.** You can now name and pin any combination of the sidebar filters and re-apply it with a single click — the app's equivalent of Lightroom's Smart Collections. A new **SAVED SEARCHES** panel sits at the bottom of the filter rail (beneath FOLDERS / TAGS / CAMERAS / DATE, just above *Clear filters*); its **Save** action captures the *current* filter state (search text, status, minimum rating, camera, selected tags, date range, the selected folder-tree scope, *show only available*, and *only unreviewed*) into a named entry. Clicking a saved entry re-applies all of that in one shot and re-runs the catalog query, so membership is **dynamic** — newly-scanned clips that match a saved search show up automatically, and clips that no longer match drop out, without any manual bucketing. Each row carries inline **rename** and **delete** actions (delete only removes the saved filter — no videos, tags, or catalog records are touched), and the save prompt pre-fills a friendly name derived from the active filters (e.g. *“Favorite · 4★+ · drone”*) so you can usually just hit Enter. Saving under an existing name updates it rather than creating a duplicate. Under the hood this adds a `SavedSearches` table (a `SavedSearch` entity storing the criteria as a JSON blob so the shape can evolve without further migrations), an `ISavedSearchService` for upsert/rename/delete, and the criteria snapshot/restore plumbing in `MainViewModel` (tag IDs are matched against the live tag set on apply, and a saved folder scope re-selects the matching folder-tree node — gracefully clearing the scope when the drive is offline or the folder no longer exists). The panel's collapse state persists across launches via `AppSettings.SidebarSavedSearchesExpanded`.

- **Diagnostics panel for self-support and troubleshooting (Help → Diagnostics…).** A new non-modal window pairs a one-glance **environment summary** with a **live tail of the application log**, so you can diagnose a problem — or produce a paste-able bug report — without leaving the app. The environment card lists the app version, .NET runtime, OS + architecture, the active **player engine** (mpv GPU vs FFME CPU) and whether the player initialised (surfacing `App.PlayerInitError` when it didn't), whether **ffprobe** was found, and the resolved on-disk paths for the **database, thumbnails, backups, settings.json, and the log file**. Below it, a virtualised list streams recent log entries (timestamp · level · source · message, with exceptions expanded beneath in red) and updates live while the window stays open — handy for watching a scan or an update check unfold. Controls: a **level filter** (All / Info & up / Warnings & up / Errors only), a free-text **filter box**, an **auto-scroll** toggle, **Copy log**, **Open log file**, **Open folder** (reveals the current log in Explorer), **Clear** (empties the on-screen buffer only), and **Copy full report** (environment summary + visible log to the clipboard). Under the hood this adds proper **file logging**: a new `DiagnosticsLog` singleton keeps a bounded in-memory ring buffer (the panel's source) *and* writes every entry to a per-session rolling file under `%LocalAppData%\VideoArchiveManager\Logs\findthatshot-<timestamp>.log` (keeping the 10 most recent sessions, BOM-tolerant, best-effort so a locked-down profile never blocks startup). It is wired into the existing `Microsoft.Extensions.Logging` pipeline as a custom `DiagnosticsLoggerProvider` registered next to the Debug provider, so *every* `ILogger` call across the app and its services is captured with **no changes to the services themselves**. EF Core's per-statement SQL logging is lifted to *Warning* via a category filter so the log stays signal rather than a wall of `SELECT`s. The window is resolved from DI (for its log/settings/ffprobe dependencies) and shown as a single reusable non-modal instance — re-invoking the menu brings the existing window forward instead of stacking duplicates.

- **Drag-and-drop folders from Explorer onto the sidebar to add them as catalog roots — and adding a folder now scans it automatically.** Familiar behaviour from Lightroom and Bridge: you can now drag one or more folders out of Windows Explorer and drop them onto the sidebar **FOLDERS** panel to register them as scan roots — the same outcome as the existing **Add folder…** picker (File menu / the panel's **+** / **Ctrl+O**), but without opening a dialog. While a folder drag hovers the panel, a dashed amber *“Drop folders to add”* drop zone overlays the body (a non-hit-testable `FolderDropOverlay` toggled from the new `FoldersPanel_OnDragEnter` / `DragOver` / `DragLeave` / `Drop` code-behind handlers), and the cursor shows the copy affordance only when the payload actually contains at least one real directory — dragging plain files is rejected so nothing happens silently. The drop is wired through a new shared `MainViewModel.AddRootFoldersByPathsAsync(paths)` that the **Add folder…** picker now also routes through: it normalises the incoming paths, keeps only existing directories, dedupes within the dropped batch *and* against roots already in the catalog (case-insensitive), persists the additions in a single `SaveChanges`, surfaces them in the `RootFolders` collection, and rebuilds the folder tree immediately so the new roots show up in the sidebar right away. **Adding folders (whether by drop or via the picker) now kicks off a scan automatically** so the clips appear without a separate, easy-to-miss **Scan** step — the previous register-only behaviour was a usability trap. To keep adding a folder cheap, the auto-scan targets only the *newly added* roots rather than rescanning the whole library: the explicit `ScanAsync` command (Catalog → Scan / F5) was refactored to share a new `RunScanAsync(roots)` core, which the add flow invokes with just the new roots (skipped if a scan is already running, so it never stomps an in-progress scan). A single rolled-up status line reports the add result (e.g. *“Added 3 root folders (1 already existed)”*) before the scan progress takes over.

- **Number-key hotkeys for your pinned tags in review mode.** You can now bind up to ten frequently-used tags to the number keys and toggle them on/off the current clip with a single keystroke while the in-app player is open — a big throughput win when working a review queue with a stable tag vocabulary, no mouse required. Bindings are configured in **Settings → Review**: ten fixed slots map to the digits **1–9 then 0** (slot 10 sits on the `0` key at the end of the number row), each picking any tag from the catalog, with a per-row clear (×), a **Clear all**, and a **Fill from most-used** button that drops your most-attached tags into the empty slots to bootstrap the list. The mapping is persisted in `settings.json` as a new `AppSettings.PinnedTags` list (each entry carries its slot + the tag's `Name` and `Type` — the catalog's natural key — rather than a DB id, so the bindings survive a catalog restore/rebuild and degrade gracefully to a no-op if a pinned tag no longer exists). In review mode `MainWindow.Window_PreviewKeyDown` maps top-row **and** numpad digits to the matching slot and calls a new `VideoDetailViewModel.ToggleTagBySlotAsync`, which reads the live mapping from settings (so a Settings edit takes effect on the next keypress) and routes to a new `ToggleTagAsync(name, type)` — already attached → detach, otherwise create-if-needed + attach. Toggling reuses the exact same attach/detach pipeline as the editor's *Add* / chip **×** (extracted into a shared `AttachTagCoreAsync`), so the first tag on an *Unreviewed* clip still auto-promotes it to *Keep*, sidecars stay in sync, the sidebar tag picker learns brand-new tags, and the status bar reports the result. Digits are ignored while a text field has focus (typing tag names / notes / coordinates) and require no modifier, so they never interfere with normal typing; the *Fill from most-used* bootstrap is powered by a new `ITagService.GetMostUsedAsync(count)` that ranks tags by attachment count. Opt-in and empty by default.

### Changed

- **The Settings window now uses a left navigation rail instead of one long scroll.** The dialog had grown to eight stacked sections (FFMPEG, Storage, Catalog backup, Sidecar files, Map, Playback, Review hotkeys, Scanning) in a single tall `ScrollViewer`, which was getting unwieldy to scan. It's now a ModernWpf `NavigationView` (Windows 11 Settings-style) with four categories — **General** (scanning / exclusions), **Library & Storage** (storage paths, sidecar files, catalog backup), **Playback & Map** (FFmpeg paths, proxy preference, telemetry overlay, flight paths), and **Review** (pinned-tag hotkeys) — each its own scrollable pane. All four panes stay resident and bound to the single `SettingsViewModel` (the code-behind toggles pane `Visibility` on nav selection rather than swapping a `Frame`), so the global **Save** / **Cancel** footer still commits every pane at once and an unsaved edit survives navigating between categories. No settings or bindings changed — purely a layout reorganisation.

### Fixed

- **The splash screen now has a crisp 1px hairline border and its artwork is clipped to the rounded corners.** A thin semi-transparent grey stroke (`BorderBrush="#66808080"`, `BorderThickness="1"`) was added to the splash window's root `Border` to delineate the card against the desktop, with `SnapsToDevicePixels` / `UseLayoutRounding` keeping the line a clean single pixel. WPF's `Border.CornerRadius` doesn't clip its child and `ClipToBounds` only clips to the *rectangular* bounds, so the full-bleed splash image was poking square corners past the rounded card. The inner `Grid` now carries a rounded `RectangleGeometry` clip (inset 1px for the border, radius 9 = 10 − 1) so the photo follows the curve and sits just inside the new border. Values are tied to the splash's fixed 600×450 size (it isn't resizable).

## [0.9.4] - 2026-06-02

### Added

## [0.9.0] - 2026-06-01

### Added

- **DaVinci Resolve-style proxy playback in the in-app player (opt-in via *Settings → PLAYBACK → Prefer DaVinci Resolve proxies for in-app playback*, default off).** Reviewing high-bitrate 4K 60p hero clips for the sole purpose of tagging is wasted decode work — Resolve users already have a proxy ladder generated for their edit (typically 1080p ProRes Proxy / DNxHR LB / H.264 in a sibling `Proxy` folder), and that's the better source to feed our player. When the new setting is on, `VideoDetailViewModel.PlayInApp` consults a new `IProxyResolver` service (`DaVinciProxyResolver`, registered as a singleton in `App.xaml.cs`) which probes `<hero-folder>/Proxy/<same-base-name>.<any-ext>` and, when a match exists, hands the proxy path to `MediaSource` instead of the hero. The match is filename-stem only (no hard-coded extension list), so it transparently handles whichever codec the user told Resolve to produce — `.mov` for ProRes/DNxHR, `.mp4` for H.264/H.265, `.mkv` for the occasional offline workflow — without us needing a new config field every time the encoder choice changes. The resolver is intentionally orthogonal to `AppSettings.ExcludedFolderNames` (which already excludes `Proxy` from the scanner so proxies never appear as their own catalog entries): the scanner does catalog work, the resolver does direct `Directory.EnumerateFiles` probes at playback time, the two systems never coordinate. Catalog entries, thumbnails, ffprobe metadata, sidecars and the *Play externally* command all continue to point at the hero file regardless of the setting — proxy substitution lives **only** at the in-app playback chokepoint. The resolver swallows all I/O failures (offline drives, permission denied, malformed paths) and returns `null`, so callers can fall back to the hero file unconditionally — proxy resolution is opportunistic by design and a missing proxy is never an error. Defaults to `false` so a fresh install behaves predictably (the file you clicked is the file that plays); users with a Resolve workflow flip it on once and forget it. `JsonSettingsStore.MergeWithUserOverrides` carries the new field through round-trip, `SettingsViewModel.BuildSettings` carries it forward on Save so opening Settings doesn't reset it, and the Settings dialog gains a new *PLAYBACK* section between *SIDECAR FILES* and *SCANNING* with the explanatory copy spelling out exactly which side of the catalog/playback split is affected.

### Fixed

- **In-app player now decodes via GPU, so high-bitrate clips (DJI drone 4K, GoPro 60p, etc.) play at real-time instead of slow motion with a flickering Play/Pause label.** FFME was wired up with no `MediaOpening` handler, which left it on its software-only FFmpeg decode path. On heavy sources the `FrameDecodingWorker` couldn't feed frames fast enough to the `BlockRenderingWorker`; the audio clock kept running while video frames arrived late, so the picture drifted into slow motion, and the engine repeatedly toggled `MediaState` between `Play` and `Manual`/`Pause` to refill its block buffer. The `PlayPauseButton.Content` is rewritten on every `IsPlaying` / `IsPaused` / `MediaState` `PropertyChanged`, so each oscillation flipped the label — that's the rapid *Play* ↔ *Pause* flicker users saw on DJI clips. Fixed by subscribing to `Unosquare.FFME.MediaElement.MediaOpening` from `MainWindow.xaml.cs` and assigning `e.Options.VideoHardwareDevices` with a priority-ordered list — `D3D11VA` first (covers essentially every modern Windows GPU), `DXVA2` as a legacy fallback, then `CUDA` for NVIDIA systems where the dedicated driver path can outperform the generic DX route. FFME tries each device until one binds; if none work (no compatible GPU, exotic codec) FFmpeg silently falls back to software decode, so this is a strict improvement on the previous behaviour. The `MediaOpening` subscription is added/removed alongside the existing `MediaOpened` / `MediaEnded` / `PropertyChanged` / `MessageLogged` wiring in the constructor and `MainWindow_Closed` so it lives and dies with the player. Hardware-device enumeration is wrapped in a defensive `try`/`catch` — a quirky GPU driver can't block playback, worst case we land back on software decode. Related upstream context: `Sinaxxr.FFME.Windows` exposes `MediaOptions.VideoHardwareDevices` (plural) which takes a priority array, and the `HardwareDeviceInfo` collection on `StreamInfo.HardwareDevices` is enumerated lazily from `avcodec_get_hw_config()` so the empty-collection path costs nothing for codecs FFmpeg can't accelerate.

## [0.5.0] - 2026-05-31

### Fixed

- **In-app player migrated from LibVLCSharp to FFME (FFmpeg-backed WPF MediaElement).** White flashes around the video — letterbox at steady state and brief transition flashes during media swap / first-frame load — were a fundamental consequence of `LibVLCSharp.WPF`'s rendering model: the `VideoView` is an `HwndHost` wrapping a Win32 child window that VLC paints into via Direct3D11. The HwndHost surface is *opaque* to WPF — its background is whatever Win32 / VLC says it is, not what the surrounding XAML says — and the DXGI swap chain VLC writes to isn't cleared between frames, so any stretch of pixels VLC hasn't actively painted (letterbox bars, the moment between media swap and first frame, the gap during seek) inherits whatever was on the surface previously. That defaulted to white via the host window class's `COLOR_WINDOW` brush. Successive iterations of fixes (class-brush patch via `SetClassLongPtr`, `WM_PARENTNOTIFY` hook, `WM_ERASEBKGND` subclass via `SetWindowSubclass`, `RedrawWindow` flush, swapping the vout backend to `direct3d9`, sizing the player area to the video aspect via a custom `AspectRatioPanel`) successively reduced the symptom but never eliminated it because none of them could reach pixels that VLC's DXGI swap chain owned. Replacing the engine cuts the root cause: FFME's `MediaElement` is a real WPF visual that renders decoded frames to a `WriteableBitmap`. Its `Background`, `Stretch`, and z-order behave like any other WPF element. There is no `HwndHost`, no class brush, no `WM_ERASEBKGND`, no DXGI swap chain — the entire failure category is gone. Specifically: removed `LibVLCSharp.WPF` 3.9.7.1 and `VideoLAN.LibVLC.Windows` 3.0.23.1 from `VideoArchiveManager.App.csproj`; added `Sinaxxr.FFME.Windows` 8.0.361-sinaxxr.2 (an actively-maintained fork of upstream FFME that targets FFmpeg 8 — picked over upstream FFME 7.0 to keep using the FFmpeg 8 shared binaries already bundled in `tools/ffmpeg/` for FFprobe, so player and metadata pipeline share one FFmpeg). `App.xaml.cs` no longer initialises `LibVLCSharp.Shared.Core`; it points `Unosquare.FFME.Library.FFmpegDirectory` at `<install>/tools/ffmpeg/` and gates `IsPlayerAvailable` on the directory containing `avcodec-62.dll`. The DI registration of the `LibVLC` singleton is gone — FFME's `MediaElement` IS the player. `MainWindow.xaml` swaps `<vlc:VideoView>` (wrapped in our `AspectRatioPanel`) for `<ffme:MediaElement Stretch="Uniform" Background="Black" LoadedBehavior="Manual" UnloadedBehavior="Manual" ScrubbingEnabled="True" VerticalSyncEnabled="True">`. `MainWindow.xaml.cs` is rewritten to use FFME's API surface: async `Open(Uri)/Close()/Play()/Pause()/Seek(TimeSpan)`, `Position` (TimeSpan), `NaturalDuration` (TimeSpan?), `IsPlaying`, `IsSeekable`. `MediaOpened` replaces VLC's `LengthChanged`; `MediaEnded` replaces `EndReached`; `INotifyPropertyChanged` filtered to `Position`/`MediaState`/`IsPlaying`/`IsPaused` replaces VLC's `TimeChanged`/`PositionChanged`/`Playing`/`Paused`/`Stopped`; `MessageLogged` filtered to `Error` severity replaces `EncounteredError`. Deleted `Helpers/Controls/AspectRatioPanel.cs` and the now-redundant `VideoItemViewModel.AspectRatio` property — FFME's built-in `Stretch="Uniform"` does the equivalent natively because the control is a real WPF visual and no longer needs an external panel to constrain it. Added an `AfterTargets="Build"` MSBuild target to `VideoArchiveManager.App.csproj` that mirrors `tools/ffmpeg/` into the build output for local debug runs (production install layout is unchanged — `publish.ps1`'s existing copy step still owns release artifacts and its `-SkipBundleFfmpeg` switch still works because the new target skips when `IsPublishing=true`). The earlier saga of GDI / DXGI workarounds — `HwndHostBackgroundFix`'s 290 lines of P/Invoke, the 4 `HwndHostBackgroundFix.Apply` call sites, the `MediaPlayer.Vout` subscription, the `--vout=direct3d9` override, the `AspectRatioPanel` plus its viewmodel binding, and the brief detour through forcing a dark `Window.Background` on every window — was net code rot in service of a problem the wrong engine was always going to have. None of it remains.
- **Tag picker no longer adds a phantom second chip when virtualization is on.** The 0.4.1 fix for "one click → multiple chips" caught the *synchronous* `SelectionChanged` echo (the picker's own `lb.SelectedItem = null` and the `Clear`/`Add` rebuild of `FilteredTags` re-firing the event on the same call stack) but missed a *deferred* path: with `VirtualizingPanel.IsVirtualizing="True"`, the picker recycles containers asynchronously when its `ItemsSource` changes, and WPF dispatches a focus/selection restore at a lower priority that fires `SelectionChanged` *after* our `finally` block returns. By that point the guard had already been reset, so whatever tag now sat at the previously-selected index (e.g. *Birds* after clicking *AQS*, since *AQS* was excluded from the rebuilt list) got promoted to a chip too, producing impossible AND-filters with empty result sets. The handler now releases the re-entrancy guard via `Dispatcher.BeginInvoke` at `DispatcherPriority.Background` so the deferred echo arrives while the guard is still set and is ignored. One click → exactly one chip, regardless of virtualization.

### Added

- **"Open file location" in the catalog right-click menu.** Right-clicking a thumbnail in the catalog grid now offers an *Open file location* item alongside the existing *Remove from database…* — picks reveal the source video file in Windows Explorer with the file pre-selected. Reuses the existing `VideoDetailViewModel.OpenFileLocationCommand` (the same command behind the editor pane's *Open location* button), so it goes through the existing `IFileSystemService.RevealInExplorer` path. As with the Remove command, it operates on the currently selected clip — make sure the right-clicked thumbnail is actually selected first.

### Changed

- **Cleaner thumbnail card metadata.** Each catalog card now shows just resolution, camera, and tag summary (in addition to the thumbnail, duration overlay, and offline badge). Filename, status, and rating were dropped to reduce visual noise on the grid: filenames are long and decorative on a visual browser, status is best edited in the right-hand panel where it lives anyway, and ratings are typically managed in DaVinci / Lightroom / Bridge.
- **Catalog thumbnails now fill the entire space between the sidebars.** The video grid used to be a `WrapPanel` with hard-coded `Width="240"` cards, so any leftover horizontal space (often 100–200px on smaller windows) showed up as empty gutter on the right and the column count jumped between 2 and 4 in coarse steps as you resized. The grid now uses a custom `AdaptiveWrapPanel` (in `Helpers/Controls/`) that picks `columns = floor(availableWidth / MinItemWidth)` (240px minimum) and gives every cell in a row the same width so each row exactly spans the catalog column at any window size. Cards in the last partial row left-align at the same width as the rest, matching the look of standard photo grids. Thumbnail tiles also keep a true 16:9 shape as cards stretch (a small `WidthToAspectHeightConverter` binds the tile's `Height` to its own `ActualWidth × 9/16`) instead of being locked at 135px regardless of card width.
- **About dialog attribution updated to reflect the FFME migration.** *Help → About* used to credit "VLC / LibVLC (LGPLv2.1+)" alongside FFmpeg in the third-party components paragraph — accurate up to 0.4.1, stale from this release onward. The dialog now credits FFmpeg (GPLv3), FFME / Sinaxxr fork (Ms-PL), and FFmpeg.AutoGen (LGPLv3), with hyperlinks to each project page. `THIRD-PARTY-NOTICES.md` was already updated as part of the migration; this just brings the in-app About dialog in sync. The publish-script comment that lists "bundled dependencies (FFmpeg, LibVLC)" is corrected the same way.
