# Changelog

All notable changes to **Find That Shot** are recorded in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **Sidebar map of the selected clip's GPS location.** The right-sidebar editor's old thumbnail image has been replaced with an interactive Leaflet map on OpenStreetMap tiles, hosted in a `WebView2` via the new `LocationMapView` user control (`Helpers/Controls/LocationMapView.cs`). When the selected clip has GPS coordinates (extracted by FFprobe from QuickTime location metadata, by `DjiSrtTelemetryReader` from a sibling `.SRT`, or by the new manual picker below) the map zooms in to z=13 with a marker; otherwise it falls back to Haugesund at a wider z=11 regional view and renders the tile pane grayscale + dimmed via a `body.no-location` CSS class so the panel reads as "no GPS data" without collapsing the layout. The map page is loaded once via `CoreWebView2.NavigateToString`; subsequent selection changes call `setLocation(lat, lon, hasLocation)` over `ExecuteScriptAsync` so the marker pans instead of the whole Leaflet stack reinitialising. Coordinate updates are coalesced on `DispatcherPriority.Background` so back-to-back Lat/Lon assignments from a `Current` switch land in a single Leaflet update (no transient pan to `(newLat, oldLon)`). WebView2 init failures degrade gracefully — the map area renders empty, the "Open in map" hyperlink in the clip-info popup still works as a fallback, and clicks on the OSM attribution route through `NewWindowRequested` → `ProcessStartInfo { UseShellExecute = true }` so they open in the default browser instead of hijacking the embedded view. Browser-y affordances (default context menus, dev tools, status bar, accelerator keys, zoom UI) are all switched off on `CoreWebView2.Settings` — this is a passive preview, not a navigable surface. Default `Microsoft.Web.WebView2` package (latest stable for Evergreen Runtime 148) is added to `VideoArchiveManager.App.csproj`.
- **Manual GPS location picker.** Clips without embedded GPS can now be geotagged from the sidebar editor. A *Set location* primary CTA appears immediately under the map whenever the clip has no GPS, and clips that already have GPS get an inline *Edit location* / *Clear* / *Copy* action row in the same slot. Clicking *Set location* (or *Edit location*) enters pick mode, swapping the action row for a coords `TextBox` plus *Cancel* / *Save*. While picking, the embedded Leaflet page installs a `map.on('click', ...)` handler that drops a marker at the clicked point and posts `{type:"pick", lat, lon}` back to the host via `window.chrome.webview.postMessage(...)`; the host parses it inside a defensive `try`/`catch` (validates `type`, finite doubles, lat/lon range), raises `LocationPicked` on the control, and `MainWindow.xaml.cs` forwards into `VideoDetailViewModel.ApplyPickedLocation` which fills both `PendingLatitude` / `PendingLongitude` and the `TextBox`-bound `PendingCoordsText` (invariant-culture `"60.482, 6.902"` formatting, matches `VideoItemViewModel.GpsText` so paste round-trips cleanly). Coordinate text can also be typed or pasted directly — the parser accepts comma-separated invariant-culture decimals and range-clamps to `[-90,90] x [-180,180]`; invalid input nulls `Pending*` so the *Save* button's `CanExecute` (via `[NotifyCanExecuteChangedFor(nameof(SaveLocationCommand))]` on both `[ObservableProperty]`s) stays false. `SaveLocationAsync` writes to `VideoItem.GpsLatitude` / `GpsLongitude`, calls `IReverseGeocodingService.LookupAsync` to refresh `LocationText` (reusing the same Nominatim service + on-disk geocode cache the *Catalog → Fill missing locations from GPS…* command already exercises — failures are non-fatal, the GPS save still succeeds and `LastSaveStatus` reports `geocode failed` / `geocode: no match`), mirrors the new values onto `Current.Model` plus a new `VideoItemViewModel.RefreshLocation()` helper that raises `PropertyChanged` for all GPS- and location-derived getters, and writes a JSON sidecar via the existing `WriteSidecarStatusAsync` path. *Clear* nulls both fields but leaves `LocationText` alone on purpose — that field can be folder-derived rather than GPS-derived, so blanking it would discard user-visible context unrelated to the pin. Selection switches mid-pick (via `OnCurrentChanged`) and entering review mode (via `OnIsPlayerVisibleChanged`) both auto-cancel any in-flight pick. The action row deliberately lives *below* the map rather than overlaid on top: WebView2's hosted Chromium HWND paints above all WPF content in its rectangle (the classic WPF "airspace" issue), so WPF buttons drawn over the map would be invisible / unclickable; the map remains the primary input surface for picking (its JS click handler is unaffected by airspace) while the chrome around it stays in plain WPF.
- **Catalog right-click → *Play externally*.** Added between *Show clip info…* and *Open file location* in the catalog grid's `ContextMenu`, wired to the existing `Detail.PlayCommand`. Mirrors the `Open file location` discoverability — the same shell-associated default media player the old sidebar *Play externally* button used. The sidebar action row is now just *Play in app* + *Open location*; *Play externally* lives on the menu since it's a secondary action.

### Changed

- **Rescan policy for GPS: embedded sources always win.** `VideoScannerService` previously used `existing.GpsLatitude ??= gpsLat;` on the Updated branch (first-writer wins), which would have stranded user-entered coordinates if the user then manually re-typed GPS while a real extractor source still existed. The branch is now `if (gpsLat.HasValue) existing.GpsLatitude = gpsLat;` — extractors (FFprobe QuickTime location, DJI `.SRT` telemetry) always overwrite when they return a value, and never null out when they don't. This is what makes the manual picker safe: user-entered coordinates survive every rescan that doesn't find embedded GPS for that clip, but get replaced the moment a real GPS source appears. No schema change.
- **Right-sidebar editor cleanup: filename + full-path block removed.** The catalog card the user clicked already shows the filename, the clip-info popup (Alt+Enter / catalog right-click → *Show clip info…*) carries the full path, and the sidebar *Open location* button reveals the file in Explorer — so the wrapping path header at the top of the sidebar was redundant clutter competing with the new map. The sidebar now reads: map (always) → action buttons (*Play in app* + *Open location*) → TAGS / NOTES / STATUS / RATING / Save changes.
- **Tag chip close glyphs render correctly again.** Both the active-tag-filter chips in the rail and the per-clip tag chips in the editor used `helpers:Theme.Icon="&#x00D7;"` (literal `×` multiplication sign), but the icon font (Segoe Fluent Icons) has no glyph at `U+00D7`, so the chip's close button rendered as a system-font tofu box. Swapped to `helpers:Theme.Icon="{StaticResource Icon.Cancel}"` (`U+E711`) — the canonical small-X glyph already used across the icon set. The doc comment in `Resources/Components/TagChip.xaml` was updated to match.

## [0.9.1] - 2026-06-01

### Added

- **PROXY badge in the player toolbar tells you at a glance whether you're watching the original clip or the substituted proxy.** Pairs with the proxy-playback setting introduced in 0.9.0: a small amber-bordered pill labelled `PROXY` (Consolas / 11px / SemiBold on `App.Accent.Subtle` background, sharing the chrome family with the *Update available* status-bar pill and the *Primary* button) floats at the left edge of the player's transport row whenever `VideoDetailViewModel.IsPlayingProxy` is `true`, and disappears entirely (no layout reservation, no greyed-out placeholder) when the player is back on the hero file. The transport row's `WrapPanel` was wrapped in a `Grid` so the chip and the centered button cluster occupy the same layout cell: `HorizontalAlignment="Left"` on the chip + `HorizontalAlignment="Center"` on the WrapPanel keeps the buttons visually centered to the full toolbar width regardless of whether the chip is present, avoiding the layout jump that a `DockPanel`/`StackPanel` reservation would have introduced.

### Fixed

- **Folder tree video counts no longer hide behind the overlay scrollbar.** The custom `App.TreeViewItem` row template reserved only 6px of right padding inside its `Bd` `Border`, but the rail's overlay scrollbar is 12px wide — so as soon as the FOLDERS tree grew tall enough to scroll, the scrollbar painted over the right-aligned `VideoCount` column on each row, leaving rows looking like they had no count (or worse, a partial count). Bumped the row's right padding to 16px to clear the scrollbar with a 4px buffer; left padding stays at 2px so the chevron + folder glyph still sit flush at the row's left edge. Pure cosmetic XAML change — no behaviour, layout, or selection logic affected.

## [0.9.0] - 2026-06-01

### Added

- **DaVinci Resolve-style proxy playback in the in-app player (opt-in via *Settings → PLAYBACK → Prefer DaVinci Resolve proxies for in-app playback*, default off).** Reviewing high-bitrate 4K 60p hero clips for the sole purpose of tagging is wasted decode work — Resolve users already have a proxy ladder generated for their edit (typically 1080p ProRes Proxy / DNxHR LB / H.264 in a sibling `Proxy` folder), and that's the better source to feed our player. When the new setting is on, `VideoDetailViewModel.PlayInApp` consults a new `IProxyResolver` service (`DaVinciProxyResolver`, registered as a singleton in `App.xaml.cs`) which probes `<hero-folder>/Proxy/<same-base-name>.<any-ext>` and, when a match exists, hands the proxy path to `MediaSource` instead of the hero. The match is filename-stem only (no hard-coded extension list), so it transparently handles whichever codec the user told Resolve to produce — `.mov` for ProRes/DNxHR, `.mp4` for H.264/H.265, `.mkv` for the occasional offline workflow — without us needing a new config field every time the encoder choice changes. The resolver is intentionally orthogonal to `AppSettings.ExcludedFolderNames` (which already excludes `Proxy` from the scanner so proxies never appear as their own catalog entries): the scanner does catalog work, the resolver does direct `Directory.EnumerateFiles` probes at playback time, the two systems never coordinate. Catalog entries, thumbnails, ffprobe metadata, sidecars and the *Play externally* command all continue to point at the hero file regardless of the setting — proxy substitution lives **only** at the in-app playback chokepoint. The resolver swallows all I/O failures (offline drives, permission denied, malformed paths) and returns `null`, so callers can fall back to the hero file unconditionally — proxy resolution is opportunistic by design and a missing proxy is never an error. Defaults to `false` so a fresh install behaves predictably (the file you clicked is the file that plays); users with a Resolve workflow flip it on once and forget it. `JsonSettingsStore.MergeWithUserOverrides` carries the new field through round-trip, `SettingsViewModel.BuildSettings` carries it forward on Save so opening Settings doesn't reset it, and the Settings dialog gains a new *PLAYBACK* section between *SIDECAR FILES* and *SCANNING* with the explanatory copy spelling out exactly which side of the catalog/playback split is affected.

### Fixed

- **In-app player now decodes via GPU, so high-bitrate clips (DJI drone 4K, GoPro 60p, etc.) play at real-time instead of slow motion with a flickering Play/Pause label.** FFME was wired up with no `MediaOpening` handler, which left it on its software-only FFmpeg decode path. On heavy sources the `FrameDecodingWorker` couldn't feed frames fast enough to the `BlockRenderingWorker`; the audio clock kept running while video frames arrived late, so the picture drifted into slow motion, and the engine repeatedly toggled `MediaState` between `Play` and `Manual`/`Pause` to refill its block buffer. The `PlayPauseButton.Content` is rewritten on every `IsPlaying` / `IsPaused` / `MediaState` `PropertyChanged`, so each oscillation flipped the label — that's the rapid *Play* ↔ *Pause* flicker users saw on DJI clips. Fixed by subscribing to `Unosquare.FFME.MediaElement.MediaOpening` from `MainWindow.xaml.cs` and assigning `e.Options.VideoHardwareDevices` with a priority-ordered list — `D3D11VA` first (covers essentially every modern Windows GPU), `DXVA2` as a legacy fallback, then `CUDA` for NVIDIA systems where the dedicated driver path can outperform the generic DX route. FFME tries each device until one binds; if none work (no compatible GPU, exotic codec) FFmpeg silently falls back to software decode, so this is a strict improvement on the previous behaviour. The `MediaOpening` subscription is added/removed alongside the existing `MediaOpened` / `MediaEnded` / `PropertyChanged` / `MessageLogged` wiring in the constructor and `MainWindow_Closed` so it lives and dies with the player. Hardware-device enumeration is wrapped in a defensive `try`/`catch` — a quirky GPU driver can't block playback, worst case we land back on software decode. Related upstream context: `Sinaxxr.FFME.Windows` exposes `MediaOptions.VideoHardwareDevices` (plural) which takes a priority array, and the `HardwareDeviceInfo` collection on `StreamInfo.HardwareDevices` is enumerated lazily from `avcodec_get_hw_config()` so the empty-collection path costs nothing for codecs FFmpeg can't accelerate.

## [0.8.0] - 2026-06-01

### Added

- **Resizable sidebar.** The left rail (FOLDERS / TAGS / CAMERAS) used to be locked at 260px, so long folder paths like `20250712 - Torvastad - Promo (Final cut)` truncated mid-name and there was no way to recover the rest of the text without a tooltip hover. A new 5px `GridSplitter` (`Resources/Components/GridSplitter.xaml` → `App.GridSplitter`) sits between the sidebar and the catalog grid; drag it to size the rail anywhere between 200px and 600px (clamped at the column level via `MinWidth` / `MaxWidth`, *and* defensively in `JsonSettingsStore.MergeWithUserOverrides` so a hand-edited `settings.json` can't shove the panel off-screen). The splitter paints a single 1px `App.Border.Subtle` hairline at rest, brightens to `App.Accent.Subtle` and grows to 3px on hover, and switches to solid `App.Accent` while dragging — visual feedback the surface is interactive without screaming for attention. Cursor flips to size-west-east anywhere across the 5px hit area, not just the 1px line. Width persists across launches via a new `AppSettings.SidebarWidth` field (default 260): `MainWindow.xaml.cs` reads it on the constructor before caching `_normalLeftWidth` so review mode round-trips to the user's drag width rather than the XAML default; `MainViewModel.PersistSidebarWidthAsync` is invoked on `Thumb.DragCompleted` (not on every layout pass — drag-completion is the explicit signal of user intent and gates the disk write to one round trip per drag). The new splitter column is collapsed to width 0 in review mode so the 5px hairline doesn't float as dead chrome between the player and the editor; `_normalSplitterWidth` is snapshotted on the way in and restored on the way out, mirroring the existing pattern for `_normalLeftWidth` / `_normalListWidth` / `_normalEditorWidth`. Trailing 1px sidebar/editor separators were left in place; together with the splitter line they form a single visually-quiet vertical rule between the rail and the grid.
- **Lightroom-style collapsible sidebar panels.** *FOLDERS*, *TAGS*, and *CAMERAS* are now wrapped in a custom `Expander` (`Resources/Components/Expander.xaml` → `App.Expander`) so users can collapse panels they aren't using and free up rail height for whichever one they're working with — the same affordance Lightroom / Bridge / Premiere users have muscle memory for. Click anywhere along the chevron + section-label band to toggle (the entire band IS a `ToggleButton` via the `App.Expander.HeaderToggle` template, not just the chevron); the chevron animates 90° between collapsed and expanded states with the same `CubicEase EaseOut` 120ms motion the folder tree uses, so the rail reads as one visual family. Trailing controls — the *Add folder* (+) on FOLDERS and the *Clear* ghost buttons on TAGS / CAMERAS — live in `Expander.Tag`, which the template routes to a content presenter *outside* the toggle hit-area so clicking them doesn't accidentally collapse the section. Hover lights up the header band with `App.Background.Hover` (4px-radius pill), matching the folder-tree row treatment from `App.TreeViewItem`.
  - **State persists across launches** via three new bools on `AppSettings` (`SidebarFoldersExpanded` / `SidebarTagsExpanded` / `SidebarCamerasExpanded`, all default `true` so a fresh install opens with everything visible) and three matching `[ObservableProperty]`s on `MainViewModel` (`IsFoldersPanelExpanded` etc.), TwoWay-bound to each `Expander.IsExpanded`. `MainViewModel.LoadSidebarPanelStateFromSettings` is called once at the top of `InitializeAsync` — *before* `ReloadFiltersAsync` and `SearchAsync` — so the rail starts in the user's last-saved shape rather than briefly flashing all-expanded and snapping shut. A `_isLoadingSidebarState` guard around the load skips the `OnIsXxxPanelExpandedChanged` write-back on initial hydration so we don't bounce the freshly-loaded values straight back to disk.
  - **Filter state survives collapse.** Selected tag chips, the camera selection, and the folder-tree selection are real catalog filters; collapsing a panel only hides the controls, it doesn't drop the active filters. Reopening the panel reveals everything unchanged. (The two view-mode `CheckBox`es — *Show only available files* / *Show only unreviewed* — are deliberately outside every expander so they remain visible regardless, since they're general view modes not section content.)
  - **Proportional rail layout.** The rail's outer `StackPanel` (which laid every panel out at its natural height regardless of free space) has been replaced with a 4-row `Grid` whose first three rows alternate between `1*` and `Auto` based on the matching `IsXxxPanelExpanded` flag (driven by a new `BoolToGridLengthConverter` registered as `BoolToStarOrAuto` in `Resources/Converters.xaml`). So when the user collapses TAGS + CAMERAS, FOLDERS gets *all* the freed vertical height — the actual Lightroom feel they were after — rather than the panel staying its natural ~320px and leaving the rest of the rail empty. Three expanded panels split rail height evenly (1/3 each); two expanded share 1/2 each; one expanded takes everything. The bottom row stays `Auto` for the two general-mode `CheckBox`es. To make this work end-to-end:
    - **`App.Expander` template's body row is now `Height="*"`** (was `Auto`) so the body fills its parent `Expander`'s allocated height. Combined with the style now setting `VerticalAlignment="Stretch"` and `VerticalContentAlignment="Stretch"`, the whole expander cooperates with the parent Grid's star sizing instead of fighting it. Collapsed expanders still measure to header-only height because the parent row is `Auto` in that case (the `*` body row can't claim space its parent doesn't have).
    - **FOLDERS body** is now a 2-row Grid (`Auto` empty-state + `*` tree) so the `TreeView` stretches into all available rail height. The previous `MaxHeight="320"` constant on the tree was dropped — star sizing is now what governs height. The tree's own internal `ScrollViewer` (already opted in via `App.TreeView` style) handles overflow when there are more folders than the granted height can show.
    - **TAGS body** is a 3-row Grid (`Auto` chips + `Auto` filter input + `*` picker list); the picker grew a `MinHeight="120"` so it stays usable in the 3-panel-expanded short-window case. Previous `Height="180"` constant gone.
    - **CAMERAS body** is the camera `ListBox` directly, with `MinHeight="100"` and `VerticalAlignment="Stretch"`. Previous `Height="120"` constant gone.
    - **No outer rail `ScrollViewer`.** Removed. Star sizing inside a `ScrollViewer` is meaningless (the `ScrollViewer` offers infinite vertical space, so `*` has nothing to share against). Each panel relies on its inner `TreeView` / `ListBox` for content scrolling — which is the Lightroom pattern anyway. The two view-mode `CheckBox`es always render at natural height in the bottom Auto row regardless of what's expanded above.
  - **Solo mode** (right-click a header to auto-collapse the others) is still deferred — it's a UX layer on top of the now-functioning proportional layout, not part of the layout itself, so it can land without further structural rework.
- **`SettingsViewModel.BuildSettings()` now carries the four new sidebar-layout fields** (`SidebarWidth`, `SidebarFoldersExpanded`, `SidebarTagsExpanded`, `SidebarCamerasExpanded`) forward from `_store.Current` when the user clicks *Save* in the Settings dialog. Without this, opening Settings → Save would reset the rail to defaults on the next launch — `BuildSettings` constructs a fresh `AppSettings` and any field not explicitly carried over takes the property default. Same pattern the existing `PageSize` / `SupportedExtensions` lines use.

### Fixed

- **Catalog "No clips match" empty state's *Clear filters* CTA now actually receives clicks.** The overlay sits in the same `Grid` cell as the catalog `ListBox` (so it visually fills the empty area without disturbing the layout). The `ListBox` has `Background="Transparent"`, which IS hit-test visible in WPF — only `Background="{x:Null}"` is non-hit-testable — and because the `ListBox` is later in document order than the `EmptyState`, it was painted on top and silently swallowed every click aimed at the centred *Clear filters* button. The empty state was visible, the button was visible, but clicking it did nothing because the click never reached the button. Set `Panel.ZIndex="1"` on the `EmptyState` so it lifts above the `ListBox` regardless of document order; the `Visibility="{Binding Videos.Count, Converter={StaticResource EmptyToVisibility}}"` binding still hides the overlay when there are clips, so this only changes hit-test behaviour in the zero-results case where the overlay is the only thing the user *should* be able to interact with anyway. Latent bug since 0.6.0 (when the EmptyState was introduced); surfaced after 0.7.0's filter UX got more powerful (e.g. clicking a camera / tag chip combination that returns nothing) and users started actually hitting the dead button.

## [0.7.0] - 2026-06-01

### Added

- **Lightroom-style folder tree in the sidebar.** The flat *ROOT FOLDERS* `ListBox` (registered roots only, no way to drill in) has been replaced with a hierarchical *FOLDERS* `TreeView` that mirrors the on-disk layout of the catalogued footage — drive → registered root → subfolder → subfolder, exactly the panel shape Lightroom / Bridge / Premiere users already have muscle memory for. Selecting *any* node filters the catalog to clips under that path (uses the existing `SearchQuery.RootFolderPath` prefix match), so the rail collapses from a "remove root folder" toolbar into a fast spatial navigator: drives across the top show recursive video counts, registered roots sit underneath, and every subfolder the scanner discovered is reachable in two clicks. Implementation details:
  - **`FolderNode` view-model** (`ViewModels/FolderNode.cs`): one node per folder with `Name` / `FullPath` / `IsDriveRoot` / `IsRegisteredRoot` / `RootFolder?` / `OwnCount` / `VideoCount` / `IsExpanded` / `IsSelected` / `Children`. `IsExpanded` and `IsSelected` are observable so tree state survives rebuilds via TwoWay bindings on the `TreeViewItem` container style — no manual visual-tree walking.
  - **Tree builder** (`MainViewModel.RebuildFolderTreeAsync`): one `SELECT DISTINCT FolderPath, COUNT(*) FROM VideoItems GROUP BY FolderPath` against the catalog, one snapshot of registered roots, then an in-memory walk that anchors each video folder to its **longest matching registered-root prefix** (so a clip under `E:\Footage\Personal` attaches to that root rather than `E:\Footage`). Drives are synthetic top-level nodes labelled the way Windows Explorer labels them — `DriveInfo.VolumeLabel` when set (`Windows (C:)`, `MediaExtension I (E:)`), otherwise a `DriveType`-based default (`Local Disk (D:)`, `Removable Disk (F:)`, `CD/DVD Drive`, `Network Drive`, `RAM Disk`), with a bare-path fallback for anything `DriveInfo` can't reason about (unresolvable UNC shares). Counts are rolled up post-order so every parent shows its recursive total. Runs from `ReloadFiltersAsync`, so every existing refresh path (initial load, post-scan, remove-folder, bulk-edit, remove-offline) keeps the tree in sync automatically.
  - **Selection / expansion persistence**: before rebuilding, the VM snapshots `{ FullPath of every expanded node }` and the current selection's `FullPath`, then reconciles against the new tree by path. Restoring selection also expands every ancestor on the path so a deep node inside a previously-collapsed branch becomes visible (not just selected in memory).
  - **Path-prefix normalisation**: `SearchQuery.RootFolderPath` is now built via `EnsureTrailingSeparator` so `E:\Footage` no longer accidentally matches `E:\Footage_v2`. The old `SelectedRootFolder.Path` path didn't normalise and had a latent false-prefix bug — fixed in passing.
  - **Custom container chrome** (`Resources/Components/FolderTree.xaml`): replaces the dated default WPF `TreeViewItem` template with a 4px-radius pill on `App.Background.Hover` / `App.Accent.Subtle` for hover / selection, a 9px Fluent chevron that animates 90° between collapsed and expanded states (`CubicEase EaseOut`, 120ms), a 16px folder glyph (`Icon.FolderOpen` + accent for drives and registered roots, `Icon.Folder` + secondary for subfolders), a `SemiBold` drive label / `Medium` root label / regular subfolder label hierarchy, and a right-aligned mono recursive count that collapses entirely when 0 so empty subtrees stay visually quiet. `IsExpanded` and `IsSelected` are TwoWay-bound to the data item on the container style itself so application sites only need to set `ItemsSource` / `ItemTemplate`.
  - **Right-click context menu** on the tree: *Remove folder…* (gated on `MainViewModel.CanRemoveSelectedFolder`, which is `true` only when the picked node maps to a `RootFolder` entity — drives and intermediate folders can't be removed) and *Add folder…* (mirrors `Ctrl+O`). Replaces the bottom-of-sidebar *Remove selected* destructive button.
  - **Inline *Add folder* affordance**: a small ghost-styled `+` button sits flush right of the *FOLDERS* section header so growing the catalog doesn't require opening the *File* menu. Hidden in the zero-folders state where the existing `EmptyState` card carries the primary CTA.
- **Sidebar *Tags* and *Cameras* lists now share their row chrome with the folder tree.** Previously the two `ListBox` filters in the rail used the default `ListBoxItem` look (14px text, system-Highlight blue selection, no rounded corners) which made them visually heavier than the new tree above them. Added `App.ListBox` / `App.ListBoxItem` in `Resources/Components/Lists.xaml`: 12px text on `App.FontFamily`, 4px-radius hover/select pill, `App.Background.Hover` hover, `App.Accent.Subtle` selection — pulled directly from the same tokens `App.TreeViewItem` uses. The three rail panels (Folders / Tags / Cameras) now read as one visual family. Applied via explicit `Style="{StaticResource App.ListBox}"` so the catalog grid `ListBox` (which carries its own card-shaped `App.VideoCard.ItemStyle`) is unaffected.
- **Background update check + status-bar *Update available* pill.** *Help → Check for updates…* was the only path to discovery, which meant most users never knew a new release was out unless they actively looked. Replaced with a calm, non-intrusive indicator: 3 seconds after the catalog finishes loading on startup, a fire-and-forget `MainViewModel.CheckForUpdatesSilentlyAsync()` calls `IUpdateService.CheckAsync()` in the background. If GitHub returns a newer release than the running build, the VM sets `IsUpdateAvailable = true` and `AvailableUpdateVersion = <semver>` on the dispatcher thread; if anything fails (no network, GitHub rate-limited, running from `dotnet run` so `NotInstalledMode=true`, malformed response), the result is dropped silently — no popup, no error in the status text, no second-guessing whether the network is OK. Per-session de-duplicated via a `_silentUpdateCheckDone` guard so it never hits GitHub more than once per launch even if `InitializeAsync` is called twice (the manual *Help → Check for updates…* command stays unaffected and can be re-run at will). When the pill is up and the user then runs the manual command, the explicit result feeds back into `IsUpdateAvailable` / `AvailableUpdateVersion` so a yanked release or out-of-band install correctly clears or refreshes the pill instead of leaving stale state. The UI side is a single 24h pill in the status-bar grid (Col 2, between the *Scan* pill and the *Sidecars* pill — new `ColumnDefinition` added), styled as a `Button` with a custom `ControlTemplate` that mirrors the *Scan* pill's chrome family (`App.Accent.Subtle` background, `App.Accent` 1px border, `Icon.Download` glyph + *Update available · v0.7.0* label in mono). Hover swaps to filled `App.Accent` with `App.Accent.OnAccent` text via a `TextElement.Foreground` trigger on the inner `Border` (inner `TextBlock`s deliberately don't carry per-element `Foreground` so inheritance flows). Clicking the pill invokes the existing `CheckForUpdatesCommand`, which opens the full *Current vX → New vY → Download and install now?* dialog and hands off to Velopack — same flow the menu item triggers. Visibility binds `IsUpdateAvailable` via `BoolToVisibility`, so the pill is fully absent when there's nothing to act on. Tooltip explains what'll happen when clicked.
- **Tag autocomplete in the editor and Bulk Edit.** The *New tag* input next to a clip and the *Add a tag* input in *Bulk Edit* are now ModernWpfUI `AutoSuggestBox`es backed by an in-memory snapshot of the tag catalog. Typing opens a dropdown of existing tags whose names contain what you've typed (case-insensitive), filtered to the currently selected tag type (Subject / Location / Person / etc., matching the `(Name, Type)` uniqueness contract in the database) and — in the per-clip editor — excluding tags already attached to the open video. ## [0.5.0] - 2026-05-31

### Fixed

- **In-app player migrated from LibVLCSharp to FFME (FFmpeg-backed WPF MediaElement).** White flashes around the video — letterbox at steady state and brief transition flashes during media swap / first-frame load — were a fundamental consequence of `LibVLCSharp.WPF`'s rendering model: the `VideoView` is an `HwndHost` wrapping a Win32 child window that VLC paints into via Direct3D11. The HwndHost surface is *opaque* to WPF — its background is whatever Win32 / VLC says it is, not what the surrounding XAML says — and the DXGI swap chain VLC writes to isn't cleared between frames, so any stretch of pixels VLC hasn't actively painted (letterbox bars, the moment between media swap and first frame, the gap during seek) inherits whatever was on the surface previously. That defaulted to white via the host window class's `COLOR_WINDOW` brush. Successive iterations of fixes (class-brush patch via `SetClassLongPtr`, `WM_PARENTNOTIFY` hook, `WM_ERASEBKGND` subclass via `SetWindowSubclass`, `RedrawWindow` flush, swapping the vout backend to `direct3d9`, sizing the player area to the video aspect via a custom `AspectRatioPanel`) successively reduced the symptom but never eliminated it because none of them could reach pixels that VLC's DXGI swap chain owned. Replacing the engine cuts the root cause: FFME's `MediaElement` is a real WPF visual that renders decoded frames to a `WriteableBitmap`. Its `Background`, `Stretch`, and z-order behave like any other WPF element. There is no `HwndHost`, no class brush, no `WM_ERASEBKGND`, no DXGI swap chain — the entire failure category is gone. Specifically: removed `LibVLCSharp.WPF` 3.9.7.1 and `VideoLAN.LibVLC.Windows` 3.0.23.1 from `VideoArchiveManager.App.csproj`; added `Sinaxxr.FFME.Windows` 8.0.361-sinaxxr.2 (an actively-maintained fork of upstream FFME that targets FFmpeg 8 — picked over upstream FFME 7.0 to keep using the FFmpeg 8 shared binaries already bundled in `tools/ffmpeg/` for FFprobe, so player and metadata pipeline share one FFmpeg). `App.xaml.cs` no longer initialises `LibVLCSharp.Shared.Core`; it points `Unosquare.FFME.Library.FFmpegDirectory` at `<install>/tools/ffmpeg/` and gates `IsPlayerAvailable` on the directory containing `avcodec-62.dll`. The DI registration of the `LibVLC` singleton is gone — FFME's `MediaElement` IS the player. `MainWindow.xaml` swaps `<vlc:VideoView>` (wrapped in our `AspectRatioPanel`) for `<ffme:MediaElement Stretch="Uniform" Background="Black" LoadedBehavior="Manual" UnloadedBehavior="Manual" ScrubbingEnabled="True" VerticalSyncEnabled="True">`. `MainWindow.xaml.cs` is rewritten to use FFME's API surface: async `Open(Uri)/Close()/Play()/Pause()/Seek(TimeSpan)`, `Position` (TimeSpan), `NaturalDuration` (TimeSpan?), `IsPlaying`, `IsSeekable`. `MediaOpened` replaces VLC's `LengthChanged`; `MediaEnded` replaces `EndReached`; `INotifyPropertyChanged` filtered to `Position`/`MediaState`/`IsPlaying`/`IsPaused` replaces VLC's `TimeChanged`/`PositionChanged`/`Playing`/`Paused`/`Stopped`; `MessageLogged` filtered to `Error` severity replaces `EncounteredError`. Deleted `Helpers/Controls/AspectRatioPanel.cs` and the now-redundant `VideoItemViewModel.AspectRatio` property — FFME's built-in `Stretch="Uniform"` does the equivalent natively because the control is a real WPF visual and no longer needs an external panel to constrain it. Added an `AfterTargets="Build"` MSBuild target to `VideoArchiveManager.App.csproj` that mirrors `tools/ffmpeg/` into the build output for local debug runs (production install layout is unchanged — `publish.ps1`'s existing copy step still owns release artifacts and its `-SkipBundleFfmpeg` switch still works because the new target skips when `IsPublishing=true`). The earlier saga of GDI / DXGI workarounds — `HwndHostBackgroundFix`'s 290 lines of P/Invoke, the 4 `HwndHostBackgroundFix.Apply` call sites, the `MediaPlayer.Vout` subscription, the `--vout=direct3d9` override, the `AspectRatioPanel` plus its viewmodel binding, and the brief detour through forcing a dark `Window.Background` on every window — was net code rot in service of a problem the wrong engine was always going to have. None of it remains.
- **Tag picker no longer adds a phantom second chip when virtualization is on.** The 0.4.1 fix for "one click → multiple chips" caught the *synchronous* `SelectionChanged` echo (the picker's own `lb.SelectedItem = null` and the `Clear`/`Add` rebuild of `FilteredTags` re-firing the event on the same call stack) but missed a *deferred* path: with `VirtualizingPanel.IsVirtualizing="True"`, the picker recycles containers asynchronously when its `ItemsSource` changes, and WPF dispatches a focus/selection restore at a lower priority that fires `SelectionChanged` *after* our `finally` block returns. - **Clicking one tag in the sidebar picker no longer adds several chips.** The picker's `SelectionChanged` handler used to read `ListBox.SelectedItem` and call `AddTagFilterCommand`, which synchronously rebuilds `FilteredTags` (`Clear` + `Add`). The mutation re-fired `SelectionChanged` with whatever item now sat at the previously-selected index, cascading into multiple chips per click. The handler now captures the clicked tag from `e.AddedItems`, clears the selection *before* the chip-add, and guards against re-entry — one click adds exactly one chip.

## [0.4.0] - 2026-05-30

### Added

- **Press Enter to add a tag.** The "New tag" textbox in the editor now accepts Enter as a shortcut for the *Add* button. Type a tag, press Enter, and the field clears so you can immediately type the next one — much faster when entering several tags in a row. The *Add* button still works exactly as before.
- **Multi-tag filtering with live search in the sidebar.** The Tags filter in the left sidebar has been redesigned to scale to hundreds of tags:
  - **Type-to-filter input** above the tag list narrows the list as you type (case-insensitive substring match).
  - **Click a tag** to add it as a removable chip; the tag disappears from the picker list while it's selected.
  - **Multiple tag filters combine with AND** — videos must carry every selected tag to appear in the result grid. Useful for queries like *Birds* ∩ *DJI Mini5Pro*.
  - **Clear** button next to the *Tags* heading drops all chips in one click; it auto-disables when nothing is selected.
  - The catalog Clear Filters command also resets the chip selection and the filter input.
- **"Start review session" workflow.** New top-toolbar button (accent-styled, next to *Clear filters*), a matching *View → Start review session* menu item, and a *Show only unreviewed* sidebar checkbox / view-menu toggle surface clips that haven't been reviewed yet:
  - **`Ctrl+Shift+R` shortcut** triggers *Start review session* from anywhere in the main window (mirrors `Ctrl+Shift+L` for *Clear filters*).
  - **Union signal.** A clip counts as "unreviewed" if EITHER its status is still the default *Unreviewed*, OR it has no tags. Catches both forgetful workflows (status never bumped) and behavioural ones (never tagged) with one filter.
  - **One-click batch start.** *Start review session* clears all other filters (search text, status, camera, dates, tag chips, root folder), turns the unreviewed filter on, runs the search, and pre-selects the first result so you can press *Play in app* immediately. Shows *"Review session: N clip(s) waiting"* or *"… you're caught up."* in the status bar.
  - **Persistent toggle.** The sidebar *Show only unreviewed* checkbox is a normal filter — combine it with the tag chips, date pickers, camera, etc. for narrower review queues (e.g. "unreviewed DJI Mini5Pro clips from May").
  - **Implemented as a new `SearchQuery.OnlyUnreviewed` flag** in the data layer, applied in `SearchService` as `Status == Unreviewed OR !Tags.Any()`.

### Changed

- **Status now auto-promotes from *Unreviewed* to *Keep* on the first tag.** Adding the first tag to a clip whose status is still the default *Unreviewed* now silently bumps it to *Keep* (status bar shows *"Tag added · status → Keep · …"* so you notice the first time). Removes one chore from the review loop and keeps the new *Show only unreviewed* filter honest over time — no more reviewed-but-still-listed clips because you forgot to flip the dropdown.
- **Tag filter is now multi-select.** The sidebar's *Tags* list used to be a single-select list that scrolled through every tag in the catalog. Replaced with the chip + search pattern above. The hidden top-bar *Tag* dropdown (which used `SelectedTagFilter`) remains commented out in the XAML for reference but is no longer wired up — the sidebar serves this purpose and scales better.

### Fixed

- **Removed the thin light borders around the in-app video player.** The player column and toolbar were drawing 1px chrome lines (`SystemControlForegroundBaseLowBrush`) on the sides of the video and between the video and the transport controls. Dropped the borders so review mode shows a clean, edge-to-edge frame against the black backdrop.

## [0.2.0] - 2026-05-29

First public release.

### Added

- **Catalog backup & restore** — automatic timestamped backups of `catalog.db` on startup, on-demand backups from Settings, configurable retention, staged restore via an app-restart so the live database is never modified while in use.
- **Sidecar JSON files (opt-in)** — write `.findthatshot.json` next to source videos so tags / ratings / notes / status travel with the footage. Atomic writes, graceful handling of read-only / offline drives, live *Sidecars: ON/OFF* indicator in the status bar, granular per-save status messages.
- **In-app update checker** — *Help → Check for updates…* looks for newer GitHub Releases and applies them via Velopack. Only takes effect when running from an installed build (`Setup.exe`).
- **Bundled FFmpeg** — `ffmpeg.exe` and `ffprobe.exe` (Gyan.dev "full" build) ship inside the installer for zero-config metadata extraction and thumbnail generation.
- **Video location services** — GPS extraction from supported video formats with reverse-geocoding via OpenStreetMap's Nominatim service. Bulk *Fill missing locations from GPS…* command on the Catalog menu.
- **Catalog cleanup** — remove individual videos, offline videos, or entire root folders (catalog only — source video files are never touched).
- **Bulk edit** — apply status, rating, append notes, or add a tag to multiple selected videos at once.

### Changed

- Responsive default window size; date pickers and the *Play externally* button are no longer clipped at common screen widths.
